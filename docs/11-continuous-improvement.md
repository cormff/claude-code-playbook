# 11 — Continuous Improvement

Every mechanism in this playbook — skills, memory, the vault, workflow discipline — is static on its own. The real payoff comes from these compounding over time into something shaped by how you actually work. Four habits drive that loop.

## Catch feedback moments

A correction ("don't do it that way") is a signal, but so is a confirmation ("yes, exactly that") — learning should come from validated decisions, not only from mistakes. Whichever it is, don't let it apply for just that one session: write it into the relevant `feedback` memory, or (if it's a repeatable behavior) into the relevant skill file, so it doesn't have to be repeated.

## Review skills periodically

A bounded but fully-authorized review pass ("read every skill, find and fix duplication and format issues") tends to surface real duplication and real mistakes far more effectively than reviewing skills one at a time as you happen to touch them. This is the one practical mechanism that keeps a skill set from silently bloating and drifting inconsistent over time.

## Keep CLAUDE.md / project instructions current

CLAUDE.md carries rules that can't be derived from the code but need to apply every session — conventions, architectural boundaries, forbidden operations. Left stale, either the same mistake gets corrected over and over, or Claude acts without ever knowing the rule existed. Unlike memory (cross-session, personal) or the vault (project content), CLAUDE.md lives in the repo itself, so it applies for the whole team, not just one person's sessions.

## The compounding effect

Memory, skills, CLAUDE.md, and a vault are each a small investment individually, but accumulated over time they produce something close to a personalized assistant — one where a new session doesn't start from zero, because it already knows your preferences and the project's actual state. That accumulation isn't automatic, though: a built mechanism (a skill, a rule, a habit) that never gets triggered wastes its own investment (see [03](03-writing-effective-skills.md#the-known-trap-a-written-skill-that-never-gets-used) and [08](08-project-development-lifecycle.md#keeping-the-vault-current-as-a-habit-not-an-afterthought)). The real discipline isn't building these mechanisms — it's *remembering to use them*.
